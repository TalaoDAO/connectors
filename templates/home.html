Interoperability through Identified and Accountable AI Agents

For AI agents to operate safely and effectively across ecosystems, they require interoperable, verifiable identities anchored in real accountability. Agents themselves do not possess legal personhood — their rights and obligations remain with the humans or organizations that operate or authorize them. An attached digital identity serves as the technical foundation of trust: a verifiable, portable, and discoverable credential layer that links every agent action back to its accountable origin.

What It Means

Each agent operates under its own cryptographic identifier and verifiable credentials, compliant with DID and Verifiable Credential Data Model (VCDM) 2.0 standards. These credentials express delegated authority — not independent personhood. The agent can sign and transact autonomously, yet every signature, token, or proof is anchored to the party that authorized it.

Why It Matters

Interoperability is impossible without clear provenance and consistent accountability semantics. Attached identity keeps responsibility traceable while enabling cross-vendor, cross-network interaction. It allows ecosystems to trust agent behaviors through verifiable data, not platform lock-in.

Delegation, Oversight, and Revocation

Delegation is expressed through Verifiable Credentials (Attestations of Delegated Authority) combined with machine-readable capability manifests. These documents define who empowered the agent, for what purpose, and within which boundaries. Oversight services can continuously monitor or revoke these credentials to enforce safety, compliance, and adaptive risk control.

From Identity to an Accountability Fabric

True interoperability is more than resolving a DID — it requires an accountability fabric that connects identifiers, digital credentials, and policies across domains. Wallet4Agent aligns with A2A and MCP protocols to ensure agents can prove their provenance, delegate rights, and exchange trust signals in a verifiable, standard-based way. This creates a global substrate for responsible, cross-ecosystem AI collaboration.

The agent’s identifier is the verifiable link in this chain — not a legal person, but a reliably identifiable actor within a wider accountability framework.

Technology Stack

All capabilities of Wallet4Agent are powered by a dedicated MCP Server that manages identity, verification, and credential issuance for AI Agents, companies, and users.

  • Security: Agent wallet HSM-backed keys in KMS • Workload-bound usage • No human access • Full audit trails .
  • Verifiable Credentials: Supports both SD-JWT VC (IETF, selective disclosure) and W3C Verifiable Credentials (JSON-LD).
  • Protocols: Implements OIDC4VCI for credential issuance and OpenID4VP for presentation and verification.
  • Decentralized Identifiers: Uses DIDs (e.g., did:web, did:jwk) to establish unique agent and organization identities.
  • Linked Verifiable Presentations (Linked VP): Enables agents to present credentials through the DID Document endpoint, simplifying trust establishment.
  • MCP Integration: All services are exposed via the Model Context Protocol (MCP) — providing a programmable interface for agents to issue, verify, and manage credentials directly. MCP server authentication through personal access token (PAT), OAuth Client Credentials flow or mTLS (option).

Get Started

It’s simple to spin up an attached identity for an agent. You can either call a public MCP tool (no bearer token required) or use the MCP Inspector to try it interactively. The MCP server is available at https://wallet4agent.com/mcp.

Option A — JSON-RPC call to /mcp (no auth)

Create an attached agent identity and wallet in one call. Provide your own credential (e.g., Google email or GitHub login) so you can later manage the wallet’s lifecycle and remain clearly identifiable as the responsible party.

Request

curl -s https://wallet4agent.com/mcp   -H "Content-Type: application/json"   -d '{
    "jsonrpc": "2.0",
    "id": 1,
    "method": "tools/call",
    "params": {
      "name": "create_identity",
      "arguments": {
        "owner_identity_provider": "google",   // one of: "google", "github", "wallet"
        "owner_login": "dev@example.com",      // Google email, GitHub login, or DID
        "agentcard_url": "https://example.com/agent-card" // optional
      }
    }
  }'

Successful response (excerpt)

{
  "jsonrpc": "2.0",
  "result": {
    "content": [{"type":"text","text":"New agent identifier and wallet created."}],
    "structuredContent": {
      "agent_id": "did:web:wallet4agent.com:<id>",
      "dev_bearer_token": "<token-for-you>",
      "agent_bearer_token": "<token-for-agent>",
      "wallet_url": "https://wallet4agent.com/<id>"
    }
  },
  "id": 1
}

You’ll receive two bearer tokens right away — one for you as developer and one for the agent. Use these to access role-based tools and manage the wallet lifecycle, keeping a clear separation between who is responsible and how the agent acts.

Option B — Use the MCP Inspector

Point the inspector to https://wallet4agent.com/mcp, discover tools, and run create_identity from the UI. Then, with your developer token, try get_data to retrieve the agent’s DID, tokens, and wallet info.

  • Guest flow: Call create_identity without a token.
  • Dev flow: Use your dev_bearer_token to call get_data.
  • Agent flow: With agent_bearer_token, request credentials via request_attestation.

Tool availability changes based on who calls the MCP server (guest, dev, or agent). The server advertises its capabilities through standard MCP routes, and tools are enforced per role, reflecting the different responsibilities of each actor.
