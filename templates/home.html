Attached Identity Model for AI Agents

AI Agents do not possess personhood or independent legal responsibility. Their identity is always attached to a responsible human or organization. The goal is not to make an agent a person, but to give it a verifiable attached identity — an aid-entity — that enables traceable actions, delegated rights, and transparent accountability across ecosystems. This design links every agent action to its originator, making responsibility auditable by design.

What it Means

The agent operates with its own cryptographic identifier and credentials, but all its authorizations are delegated from a real accountable party. It can sign, act, and interact autonomously — yet every signature is traceable to its source of mandate.

Why it Matters

This model keeps responsibility where it belongs: with people and organizations. By attaching identity instead of granting personhood, we enable trust, auditability, and compliance while preserving human accountability.

Delegation and Oversight

Delegation is formalized through Verifiable Credentials (mandates) and operational permissions (capabilities). Together they express who empowered the agent, why, and within what limits. Oversight tools can monitor and revoke these rights when behavior must be adjusted.

Supporting Responsible Autonomy

AI agents are adaptive and probabilistic. The attached identity model enables autonomy without disconnection — giving agents freedom to act, but within a framework of traceability and verifiable responsibility.

AI Agents are becoming active participants in digital ecosystems — signing transactions, exchanging data, and making autonomous decisions. Wallet4Agent enables this evolution by giving agents a decentralized identity (DID), verifiable credentials, and policy-based authorizations that comply with both decentralized and regulated identity frameworks.

By combining decentralized identity, open protocols, and verifiable credentials, Wallet4Agent creates a new trust foundation for the Agentic Web. Every interaction can be independently verified, ensuring security, transparency, and compliance with digital-identity regulations such as eIDAS v2 and the upcoming AI governance frameworks.

Technology Stack

All capabilities of Wallet4Agent are powered by a dedicated MCP Server that manages identity, verification, and credential issuance for AI Agents, companies, and users.

  • Verifiable Credentials: Supports both SD-JWT VC (IETF, selective disclosure) and W3C Verifiable Credentials (JSON-LD).
  • Protocols: Implements OIDC4VCI for credential issuance and OpenID4VP for presentation and verification.
  • Decentralized Identifiers: Uses DIDs (e.g., did:web, did:jwk) to establish unique agent and organization identities.
  • Linked Verifiable Presentations (Linked VP): Enables agents to present credentials through the DID Document endpoint, simplifying trust establishment.
  • MCP Integration: All services are exposed via the Model Context Protocol (MCP) — providing a programmable interface for agents to issue, verify, and manage credentials directly.

Get Started

It’s simple to spin up an identity for an agent. You can either call a public MCP tool (no bearer token required) or use the MCP Inspector to try it interactively. The MCP server is available at https://wallet4agent.com/mcp.

Option A — JSON-RPC call to /mcp (no auth)

Create an agent identity and wallet in one call. Provide your own credential (e.g., Google email or GitHub login) so you can later manage the wallet’s lifecycle.

Request

curl -s https://wallet4agent.com/mcp   -H "Content-Type: application/json"   -d '{
    "jsonrpc": "2.0",
    "id": 1,
    "method": "tools/call",
    "params": {
      "name": "create_identity",
      "arguments": {
        "owner_identity_provider": "google",   // one of: "google", "github", "wallet"
        "owner_login": "dev@example.com",      // Google email, GitHub login, or DID
        "agentcard_url": "https://example.com/agent-card" // optional
      }
    }
  }'

Successful response (excerpt)

{
  "jsonrpc": "2.0",
  "result": {
    "content": [{"type":"text","text":"New agent identifier and wallet created."}],
    "structuredContent": {
      "agent_id": "did:web:wallet4agent.com:<id>",
      "dev_bearer_token": "<token-for-you>",
      "agent_bearer_token": "<token-for-agent>",
      "wallet_url": "https://wallet4agent.com/<id>"
    }
  },
  "id": 1
}

You’ll receive two bearer tokens right away — one for you as developer and one for the agent. Use these to access role-based tools and manage the wallet lifecycle.

Option B — Use the MCP Inspector

Point the inspector to https://wallet4agent.com/mcp, discover tools, and run create_identity from the UI. Then, with your developer token, try get_data to retrieve the agent’s DID, tokens, and wallet info.

  • Guest flow: Call create_identity without a token.
  • Dev flow: Use your dev_bearer_token to call get_data.
  • Agent flow: With agent_bearer_token, request credentials via request_attestation.

Tool availability changes based on who calls the MCP server (guest, dev, or agent). The server advertises its capabilities through standard MCP routes, and tools are enforced per role.
