Technology Stack

All capabilities of Wallet4Agent are powered by a dedicated MCP Server that manages identity, verification, and credential issuance for AI Agents, companies, and users.

Wallet4Agent offers an extensible, standards-based stack that connects AI Agents to both regulated identity ecosystems (eIDAS v2 / EUDI Wallet, EBSI, Data Spaces) and decentralized ecosystems (DIDs, W3C Verifiable Credentials, OIDC4VC).

It focuses on keeping cryptographic keys off the Agent runtime while still allowing the Agent to request signatures, present credentials, and manage its own identity through MCP tools.

  • Security: Agent wallet HSM-backed key management & MPC-style separation between AI Agents, the Wallet4Agent backend, and underlying identity networks. Workload-bound usage • No human access • Full audit trails.
  • Verifiable Credentials: Supports SD-JWT VCs (selective disclosure) and W3C Verifiable Credentials (JSON-LD) for interoperability with EUDI Wallets, enterprise wallets, and SSI ecosystems.
  • Protocols: Implements OIDC4VCI for credential issuance and OIDC4VP for presentation and verification across ecosystems.
  • Decentralized Identifiers: Uses DIDs (e.g. did:web, did:cheqd) to establish unique agent and organization identities.
  • Linked Verifiable Presentations (Linked VP): Enables agents to expose verifiable presentations via HTTP endpoints and attach them to the DID Document endpoint, simplifying trust establishment.
  • MCP Integration & Tools: All services are exposed via the Model Context Protocol (MCP). Tools are scoped by role:
    • Guest tools: create_agent_identifier_and_wallet.
    • Dev tools: get_configuration, create_OASF, update_configuration, rotate_personal_access_token, add_authentication_key, get_attestations_of_this_wallet, delete_identity.
    • Agent tools (identity & wallet): get_this_agent_data, get_attestations_of_this_wallet, get_attestations_of_another_agent, accept_credential_offer, sign_text_message, sign_json_payload, describe_wallet4agent, explain_how_to_install_wallet4agent.
    • Agent tools (verification): start_user_verification, poll_user_verification, start_agent_authentication, poll_agent_authentication.
    Authentication for MCP server access supports Personal Access Tokens (PAT) and OAuth 2.0 Client Credentials (including private_key_jwt with JWK keys).
Example cryptographic flow

A typical flow for an Agent requesting a credential using OIDC4VCI: 

// Agent (via MCP) asks Wallet4Agent to accept a credential offer
call accept_credential_offer {
  "credential_offer_uri": "<oidc4vci_offer_url>"
}

// Wallet4Agent fetches issuer metadata & obtains a token
// It generates a proof of key ownership using the Agent's KMS key

// Issuer returns a Verifiable Credential (SD-JWT VC or VC JSON-LD)
// Wallet4Agent stores it as an attestation in the agent's wallet
The Agent never sees raw keys — only tool responses with structured results. Wallet4Agent handles KMS interactions, proofs, and VC storage.

What AI Agents can do with Wallet4Agent

Wallet4Agent lets you give AI Agents a verifiable, governed identity and wallet, so they can act on behalf of people and organizations with real-world accountability.

Each agent gets its own DID, credentials, and scoped MCP tools that enforce what it is allowed to see, say, and do.

For Developers & LLM Integrators

Using Wallet4Agent from an MCP-compatible LLM

Wallet4Agent runs as an MCP Server that exposes tools for identity, credentials, and authorization. Any MCP-compatible LLM client (ChatGPT, Claude, local MCP runners, etc.) can attach it and give agents a real-world identity and trust layer.

Option A — Guest → Agent (+ Dev)

As a guest (no auth header), you can bootstrap a new Agent identifier & wallet using: create_agent_identifier_and_wallet.

This returns:

  • An Agent identifier (DID) such as did:web:wallet4agent.com:<id> or, depending on deployment, did:cheqd:<network>:<id>.
  • A wallet URL where the Agent's DID Document and Linked VPs are published.
  • A dev_personal_access_token used to manage configuration and tokens.
  • Either an agent_personal_access_token (PAT) or OAuth 2.0 client credentials for the Agent.
Option B — Agent PAT

For rapid prototyping, use the Agent Personal Access Token (PAT) returned by create_agent_identifier_and_wallet. 

// Agent calls MCP with a PAT
Authorization: Bearer <agent_personal_access_token>

// Example MCP request
call get_this_agent_data {}
Simple to start with; suitable for sandbox environments. You can later rotate tokens or switch to OAuth2 without changing MCP tools.
Option C — OAuth 2.0 Client Credentials

For production deployments, configure the Agent as an OAuth 2.0 client (Client Credentials grant), using agent_client_id (the Agent DID) and agent_client_secret or private_key_jwt. 

// Discover token endpoint
GET https://wallet4agent.com/.well-known/openid-configuration

// Then request a token
POST /oauth/token
  grant_type=client_credentials
  client_id=<agent_identifier_did>   // e.g. did:web:... or did:cheqd:...
  client_secret=<agent_client_secret>
  resource=https://wallet4agent.com

When configured for OAuth2, the Agent first calls the AS token endpoint with its client_id and client_secret (or private_key_jwt), then uses the resulting access token in the MCP Authorization: Bearer header.

Option D — Explore everything with an MCP Inspector

Point your MCP Inspector to https://wallet4agent.com/mcp, discover the tools, and experiment:

  • Guest: Call create_agent_identifier_and_wallet to bootstrap a new agent DID (e.g. did:web or did:cheqd, depending on configuration) and its attached wallet.
  • Developer: Use your dev_personal_access_token to call get_configuration, create_OASF, update_configuration, rotate_personal_access_token, add_authentication_key, get_attestations_of_this_wallet, and delete_identity.
  • Agent: With an agent PAT or OAuth2 access token, call describe_wallet4agent, explain_how_to_install_wallet4agent, get_this_agent_data, get_attestations_of_this_wallet, get_attestations_of_another_agent, accept_credential_offer, sign_text_message, or sign_json_payload.
  • Verifier (agent verifying users or other agents): Use start_user_verification and poll_user_verification for email-based user verification, or start_agent_authentication and poll_agent_authentication for agent-to-agent verification flows.
Documentation

Tool availability and returned fields depend on who calls the MCP server (guest, developer, agent or verifier) and on the configured authentication mode (PAT vs OAuth2). The server advertises its capabilities through standard MCP routes, and enforces role-based access to keep responsibilities clearly separated.
