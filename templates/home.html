We provide Identity, Attestations and Wallets to AI Agents

AI Agents can now authenticate and act on behalf of natural persons, legal persons or other AI Agents with their own-Identity, attestations and mandates. This server exposes issuer, holder and verifier capabilities via the Model Context Protocol.

MCP info MCP tools

Our global approach: AI Agents are identities

One identity model for everyone

We consider AI Agents as first-class identities, just like natural persons and legal entities. Every identity can hold a wallet, receive credentials, present proofs, and exercise mandates (delegations) within policy.

  • Issuance: grant identity, role, or mandate credentials to people, companies, and agents.
  • Verification: request and verify proofs (e.g., profile, age, membership, compliance) with selective disclosure.
  • Lifecycle: store, update, suspend, or revoke credentials with auditability.

This unifies human–organization–agent interactions under W3C DIDs, Verifiable Credentials, and OIDC4VC flows.

What you can do today

  • Authenticate users, organizations, or other agents via wallet-based presentations.
  • Delegate authority to agents with verifiable mandates for scoped tasks.
  • Prove attributes and compliance with minimal disclosure and revocation checks.
  • Interoperate across ecosystems and wallets thanks to standards alignment.

Available MCP tools

These tools are exposed through the Model Context Protocol (MCP) interface for agents. See /mcp/info and /manifest.json for machine‑readable metadata.

request_attestation

Request the credential which is offered and store it in the wallet.
Inputs (JSON Schema) 
{
  "type": "object",
  "properties": {
    "credential_offer": {"type": "string", "description": "OIDC4VCI credential offer"},
    "agent_id": {"type": "string", "description": "Agent DID (e.g., did:web:wallet4agent:demo)"}
  },
  "required": ["agent_id", "credential_offer"]
}

create_identity

Generate a DID for the agent and create a new wallet to store agent attestations.
Inputs (JSON Schema) 
{
  "type": "object",
  "properties": {
    "did_method": {"type": "string", "enum": ["did:jwk", "did:web"], "description": "Optional DID method"},
    "agentcard_url": {"type": "string", "description": "Optional AgentCard URL"},
    "always_human_in_the_loop": {"type": "boolean"},
    "owner_identity_provider": {"type": "string", "enum": ["Google", "Github", "Data Wallet", "Test"]},
    "owner_login": {"type": "string", "description": "email/login/DID for owner"},
    "name": {"type": "string", "description": "Name of the wallet"}
  },
  "required": ["owner_identity_provider"]
}

Security & privacy

Least-privilege MCP

Only the start/poll/revoke actions are exposed to agents.

Token redaction

Raw tokens are not returned; the server exposes derived claims only.

TTL-based cleanup

Flows expire automatically in the verifier backend (Redis TTL). No data are stored on servers.
